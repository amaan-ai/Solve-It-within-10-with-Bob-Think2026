Build a Credit Card Payment Processing Application Agent Mode

Build a mock credit card payment processing application with: Backend (Java 11, Spring Boot): - REST API endpoints: - POST /api/payments/authorize (authorize a card transaction) - POST /api/payments/capture (capture an authorized transaction) - POST /api/payments/refund (refund a captured transaction) - GET /api/payments/{id} (get transaction status) - GET /api/payments/history (list recent transactions) - POST /admin/cache/clear (clear local cache) - GET /actuator/health (readiness probe) - GET /actuator/prometheus (metrics endpoint) - Use in-memory H2 database (no external DB needed) - Local Caffeine cache for transaction lookups - Simulate realistic processing delays (200-500ms) - Simulate random declines (10% of transactions) - Return realistic response codes (approved, declined, insufficient funds, expired card) - Use the test card numbers: - Visa: 4263970000005262 - MasterCard: 5425230000004415 - Amex: 374101000000608 Frontend (React): - Payment form: card number, expiry, CVV, amount - Transaction history dashboard showing recent payments - Status badges: Authorized (yellow), Captured (green), Declined (red), Refunded (gray) - Simple and clean — this is for a demo, not production Make it run with a single command: mvn spring-boot:run Do not run the application. I will run it myself. Important implementation notes: 1. `GenerationType.UUID` requires JPA 3.x — Spring Boot 2.7 ships with JPA 2.x (Hibernate 5), which does not support `GenerationType.UUID`. Use `UUID.randomUUID()` in a `@PrePersist` hook instead, or upgrade to Spring Boot 3.x. 2. Lombok requires annotation processing to be active at compile time — When Lombok is on the classpath but the compiler's annotation processor isn't wired correctly (e.g., missing `-proc:full` flag or IDE misconfiguration), all generated methods (`getters`, `setters`, `builder`, `log`) are invisible to `javac` and produce "cannot find symbol" errors. For a zero-friction demo, replacing Lombok with explicit getters/setters and a hand-written builder is more portable and avoids the dependency entirely. Use the following folder structure: ``` payment-app/ ├── src/ │ └── main/ │ ├── java/com/demo/payment/ │ │ ├── controller/ │ │ ├── service/ │ │ ├── model/ │ │ └── config/ │ └── resources/ │ ├── static/ ← React frontend │ └── application.properties ├── pom.xml └── README.md ```

Build a Citizen Service Request Agent Mode

Build a mock Citizen Service Request application for a government agency using Java 17 and Spring Boot 3. Backend (Java 17, Spring Boot 3): - REST API endpoints: - POST /api/citizens — register a new citizen (nationalId, fullName, dateOfBirth, email) - GET /api/citizens/lookup?nationalId=XXX — look up citizen by national ID - POST /api/requests — submit a service request (citizenId, serviceType, description) - GET /api/requests/{id} — get request status - PATCH /api/requests/{id}/status — officer updates request status (body: status, officer, remarks) - GET /api/requests/dashboard — returns count of requests per status - Service request types: PASSPORT_APPLICATION, BIRTH_CERTIFICATE, DRIVING_LICENCE_RENEWAL, SOCIAL_BENEFIT_CLAIM, TAX_FILING, VOTER_REGISTRATION - Business rules to enforce: - National ID must be unique and exactly 12 uppercase alphanumeric characters - Only ACTIVE citizens can submit requests - Status workflow: SUBMITTED → UNDER_REVIEW → APPROVED | REJECTED (no skipping steps) - resolvedAt is auto-set when status reaches APPROVED or REJECTED - Use in-memory H2 database — no external DB needed - Seed 3 demo citizens and 4 service requests in various states on startup Frontend (React): - Citizen registration form and national ID lookup - Service request submission form with dropdown for request type - Officer dashboard: table of all pending requests with status update controls - Status badges: Submitted (blue), Under Review (yellow), Approved (green), Rejected (red) - Simple and clean — this is for a demo, not production Make it run with a single command: mvn spring-boot:run Do not run the application.
